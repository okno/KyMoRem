# KyMoRem FAQ EN

This FAQ captures the real issues found during `0.2.0-rc2` development and
testing.

## Definitive recovery when a client does not work

1. On the server press `RILASCIA` or `Ctrl+Esc`.
2. Close every old KyMoRem client window on the target machine.
3. Restart the KyMoRem server app.
4. Regenerate the target package from the server.
5. Install/start the package on the target machine.
6. Press `AGGIORNA` on the server.
7. Check that the card is `ONLINE`.
8. Enter from the configured edge.

## Windows 7 receives no mouse or keyboard

Most common causes:

- old Windows 7 executable still open;
- wrong architecture, for example x64 on Windows 7 x86;
- firewall closed on `54865/tcp`;
- package started without `kymorem-token.txt`;
- layout saved but not refreshed.

Recommended fix:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\New-KyMoRemWin7ClientPackage.ps1 -Arch x86 -Name windows7 -Direction down -Zip
```

Copy the package to Windows 7, run `Install-Firewall-And-Start.cmd` as
Administrator, leave the black client window open, then press `AGGIORNA` on the
server.

## Secure handshake rejected

If the message says `cryptography` is required, the client is old or incomplete.
Install the rc2 package generated by the server.

If rejection repeats from the server IP, the tokens do not match. Do not type
the token manually. Regenerate the package so `kymorem-token.txt` and
`%APPDATA%\KyMoRem\config.json` stay synchronized.

## Server shows 0 clients but cards are online

Discovery and TCP health are different:

- discovery uses `54866/udp`;
- control and health use `54865/tcp`.

Wi-Fi isolation, VLANs or firewalls can block UDP while TCP still works. Press
`AGGIORNA` and use a manual IP when broadcast is unreliable.

## I moved machines but routing did not change

Correct sequence:

1. Select the client.
2. Move it on the map.
3. Press `SALVA`.
4. Press `AGGIORNA`.
5. Release with `Ctrl+Esc`.
6. Enter from the new edge.

Routing is recalculated on the next edge transition.

## I cannot reach Windows 7 from linux-iMac

Install rc2 on the server and Windows 7 client. rc2 closes the previous remote
endpoint before opening the new one; older builds could remain attached to the
Linux client.

Also verify adjacent coordinates:

```text
server     x=0 y=0
linux-iMac x=1 y=0
windows7   x=2 y=0
```

In this example, the right edge of `linux-iMac` enters `windows7`.

## I move down but the pointer does not appear on Windows 7

Check that `windows7` is below the server:

```text
server   x=0 y=0
windows7 x=0 y=1
```

Then `SALVA`, `AGGIORNA` and retry from the bottom edge. If the card is
`STANDBY`, the layout exists but the TCP client is not live.

## Infinite mouse scroll blocks everything

This happened with high-resolution mice when every wheel tick was queued. In
rc2 wheel input is coalesced, capped and cleared on release/client switch.
Update server and clients. If a backlog already exists, press `Ctrl+Esc` and
restart the affected client.

## Linux client is online but does not move

Check X11:

```bash
echo "$XDG_SESSION_TYPE"
echo "$DISPLAY"
ls -l "$HOME/.Xauthority"
command -v xdotool
xdotool getmouselocation
```

For rc2, production Linux input injection requires X11. Wayland is not a full
input-injection target.

## User passwords

KyMoRem does not use Windows, Linux, SMB, RDP or SSH passwords for normal
operation. Trust comes from the shared token, approved clients and encrypted
channel.

## Ports and logs

```text
54865/tcp  encrypted session
54866/udp  encrypted discovery
```

Server log:

```text
%APPDATA%\KyMoRem\server.log
```

Windows 7 log:

```text
kymorem-win7-client.log
```

Linux log:

```text
${XDG_RUNTIME_DIR:-/tmp/kymorem-$UID}/kymorem-client.log
```
